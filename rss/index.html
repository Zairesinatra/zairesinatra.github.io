su 和 su - 都是需要输入目标用户的密码以切换到另外的 shell 实例。那么很明显的，由于登录 root 用户就需要 root 身份密码，在多用户服务器中通过 su、su - 进行 root 访问权限意味着 root 用户密码隐私性欠佳。

sudo (superuser-do助记符) 的使用原理与上述是南辕北辙的。即通过一个配置文件(/etc/sudoers) 列出每个用户的特定操作权限。当调用时，要求验证配置文件中的用户身份密码来允许提升权限运行单个命令且不会启动一个新的 shell。

通常在 Linux 系统中非 root 用户运行sudo命令，会提示以下信息

# xxx 当前用户无法执行 sudo 命令
$ xxx is not in the sudoers file.  This incident will be reported.

那么就需要 su - 进入超级用户后给配置文件 /etc/sudoers 添加添加 xxx 权限。这里注意编辑配置文件可能操作会因权限问题被阻止，可以使用 chmod u+w /etc/sudoers 加写权利。在 nvim /etc/sudoers 完成添加 xxx ALL=(ALL)ALL 后撤销文件的写权利 chmod u-w /etc/sudoers。

su 与 su -

这里有个问题，上文中使用 su -i 是什么意思，和直接 su 有什么区别？

首先能确定的是 su 和 su - 都会切换到 root 用户。但是就环境来说，su 切换到 root 的同时，当前目录保持普通用户的不变。而 su - 可以看成用户和环境一起切为 root 身份。

/home/newghostzs

newghostzs@VM-4-10-ubuntu:~$ su -

sudo -s 与 sudo -i

和 sudo 允许以自己的用戶账户以 root 权限运行命令又不同。sudo -s 以 root 权限运行一个 shell 。而 sudo -i 不但以 root 权限运行一个 shell，还会获取 root 用户的环境。更直观来说前者在用户环境中保留 ~/.bashrc 后者在根环境忽略 ~/.bashrc。

二分搜索

如果待查找的数是整数，且知道范围，可以考虑二分查找。

  • 先设定左下标 left 与右下标 right，再计算中间下标 mid。通常考虑数组头尾。
  • 根据 arr[mid] 和 target 之间的大小进行判断，相等则直接返回下标，nums[mid] < target 则 left 右移，nums[mid] > target 则 right 左移。
function BinarySearch (array, value) {
    let left = 0, right = array.length - 1;
    while( left <= right ) {
        var mid = parseInt((left + right) / 2);
        if (value == array[mid]) {
            return mid;
        } else if (value > array[mid]) {
            left = mid + 1;
        } else if (value < array[mid]) {
            right = mid - 1;
        }
    }
    return "nonono";
}

function interpolationSearch(arr, value) {
    let low = 0, high = arr.length - 1, position = -1, delta = -1;
    while (low <= high && value >= arr[low] && value <= arr[high]) {
      delta = (value - arr[low]) / (arr[high] - arr[low]);
      position = low + Math.floor((high - low) * delta);
      if (arr[position] === value) {
        return position
      }
      if (arr[position] < value) {
        low = position + 1
      } else {
        high = position - 1
      }
    }

}

function shuffle (array) {

防抖策略(debounce)：

意义： 保证频繁触发事件时，事件回调仅被触发一次。减少请求次数、节约请求资源。

定义： 当事件被触发后，延迟n秒再执行回调，若在这n秒内事件又被触发则重新计时。——类似游戏英雄联盟的回城操作。

图例：

应用： 输入框防抖-淘宝搜索框中连续输入字符，保证仅在输入完成后才进行查询的请求。

                timer = setTimeout(() => { // 没timer
                    fn();
                }, delay);
            }

区分：

在版本 5 中，npm 引入了 package-lock.json 文件。该文件旨在跟踪被安装的每个软件包的确切版本，以便产品可以以相同的方式被 100％ 复制（即使软件包的维护者更新了软件包）。
而这也是 package.json 一直尚未解决的特殊问题。 在 package.json 中，可以使用 semver 表示法设置要升级到的版本（补丁版本或次版本），例如：

  • 如果写入的是 〜0.6.0，则只更新补丁版本：即 0.6.1 可以，但 0.7.0 不可以。
  • 如果写入的是 ^0.6.0，则要更新补丁版本和次版本：即 0.6.1、0.7.0、依此类推。
  • 如果写入的是 0.6.0，则始终使用确切的版本。

  • 那么 package-lock.json 可以固话当前安装的每个软件包版本。再运行 npm install时，npm 会使用确切的版本。
  • 同时，如果运行 npm update 时，package-lock.json 文件的依赖版本会被更新。

  • resolved：指向软件包的位置
