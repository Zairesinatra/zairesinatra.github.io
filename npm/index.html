NPM

来世不可待，往世不可追也 ——庄子 《人世间》

Ziyi Xie

Apr 25, 2020 • 4 min read

近年来 NodeJS 逐渐成为前端开发的必备环境，NPM 作为 NodeJS 自带的包管理模块，解决 JS 代码包的下载管理与关系依赖，允许用户从 NPM 服务器下载第三方包和命令行程序到本地使用，甚至开源自己的代码。现在开发中常见的主流包管理模块有 npm、yarn、bower。

初识 NPM 通常是从 CLI Command 及模块描述文件 package.json 开始，随后再逐渐深入 npmrc、npm registry、semver、npm access 之流。

CLI 命令

CLI - Commond Line Interface 对应的是 GUI (图形化介面)，CLI 通常只支持键盘录入指令 ，不支持鼠标操作。Windows 的 PowerShell 或是 Mac 的 terminal 都是CLI工具。

# 常用命令
# 安装包的名称和版本号存到生产环境 dependencies 中
npm i _module_ -S
npm install _module_ --save
# 安装包的名称和版本号存到开发环境 devDependencies 中
npm i _module_ -D
npm install _module_ --save-dev
# 全局安装
npm i _module_ -g
# 清除npm缓存
npm cache clean -f
# 查看某个 package 的最新版本|在 npm 服务器上所有发布过的版本
npm view package version|versions

存在 dependencies 中的，是在生产环境中依赖包，而 devDependencies 中储存开发环境中使用的依赖包。比如项目插件：例如 Element-UI、Echarts 这种插件要在运行中使用的，就要放在 dependencies 中( -S ) 、gulp、webpack 此类用来压缩代码，打包等需要的工具，程序实际运行的时候并不需要，就放在 devDependencies 中( -D )。

package.json 与 package-lock.json

考虑到一般不会将 node-modules 文件夹提交 Git，那么通过 npm install 安装包时，可能出现版本不兼容，以及补丁版或者次版本缺陷等问题。而这也是 package.json 的特殊弊端。在 NPM5 引入的 package-lock.json 文件可以跟踪并固化被安装的每个软件包的确切版本，以便包模块以相同的方式被 100％ 复制（即使软件包更新）。

在 package.json 中，可以使用 semver 表示法设置要升级到的版本（补丁版本或次版本）。版本号严格按照 主版本号.次版本号.修订号 格式命名，当某个版本改动比较大、并非稳定而且可能无法满足预期的兼容性需求时，会先发布一个先行版本。先行版本号可以加到“主版本号.次版本号.修订号”的后面，先加上一个连接号再加上一连串以句点分隔的标识符和版本编译信息。常见先行版本说明：内部版本(alpha)、公测版本(beta)、正式版本的候选版本rc (Release candiate)。

当 npm 包某些功能布新的版本，通常的做法是直接去修改 package.json 到指定版本。package-lock.json 文件中 integrity 用于校验软件包；resolved 指向软件包的位置。

〜0.6.0 # 更新补丁版本.即 0.6.1 可以,但 0.7.0 不可以
^0.6.0 # 更新补丁版本和次版本 0.6.1、0.7.0、依此类推
0.6.0 # 始终使用确切的版本

如果自行发布包，在修改模块后需要更新发布版本，可以通过 Semver 规范的命令来完成，避免版本混乱。

npm version patch : 升级修订版本号
npm version minor : 升级次版本号
npm version major : 升级主版本号

npmrc

.npmrc 文件是 NPM 的用户配置文件。NPM 从命令行、环境变量和 npmrc 文件中获取其配置设置。使用 npm config 相关命令可用于更新和编辑用户和全局 npmrc 文件的内容。

# 查看路径
npm config get userconfig
# 显示当前仓库源
npm config get registry
# 设置国内镜像|官方镜像
npm config set registry=https://registry.npm.taobao.org|https://registry.npmjs.org
