JWT跨域认证解决方案

前言

在项目中难免遇见跨域问题。就好比 登录界面 的业务流程分析完成后，在技术实现上需要考虑跳转与否。项目运行新端口号，服务器可能有跨域问题，选择 token 方式；如果前端与后台接口不存在跨域问题，推荐cookie和session。这里通过项目的思路，总结一下跨域的 token 以及 JWT。

正文

技术过程：

JSON Web Token（缩写 JWT）是目前最流行的跨域认证解决方案。JWT 的原理是，在确认客户端登陆到服务器认证以后，会生成一个 JSON 对象，发回给客户端。客户端此时存储此 JSON 对象。以后，用户与服务端通信的时候，都要发回这个 JSON 对象。服务器完全只靠这个对象认定用户身份。

JSON Web Token中JSON、Web都很熟悉了，那么这里说明一下Token。

Token分析

这里以 POSTMAN 接口测试工具返回登陆效果，携带的token为例，分析一下token。

可以很明显的观察到，token是一个很长的字符串，中间通过点（.）分隔成三个份。依次分为：header（头部区）、payload（负载区）、signature（签名）三部分。
即token=header.payload.signature。

  • Header-JSON对象
{"alg":"HS256","typ":"JWT"}

这是默认token的header。"alg"=>"algorithm"、"HS256"=>"HMAC SHA256"。

在上述贴图中未上线测试的项目采用的是Bearer Token (RFC 6750) 用于OAuth 2.0授权访问资源，任何Bearer Token持有者都可以无差别地用它来访问相关的资源，而无需证明持有加密key。

好了，收！有兴趣可以自行去研究其他跨域认证，如下。

  • Payload-JSON对象
    JWT规定Payload通过7个官方字段存放传递数据，同时也可以自定义增设（JWT默认不加密，不加敏感数据）：
iss (issuer)：签发人
exp (expiration time)：过期时间
sub (subject)：主题
aud (audience)：受众
nbf (Not Before)：生效时间
iat (Issued At)：签发时间
jti (JWT ID)：编号
  • Signature
    此部分设置用来防止header、payload被篡改。这里也制定了仅服务器可查询的秘钥。signature算法也为HS356。

那么通过上述的图标与解释，这个JWT在客户端接受后，存放在 Cookie 或 localStorage 中。在日后需要与服务器通讯，就可以放在 HTTP 请求的头部信息 Authoruzation 字段中（或者POST请求体内），看的具体方法在上图 Chrome 浏览器截图。

结束

虽然JWT解决方案很香，不过token一旦生成就无法在过期前终止。且JWT默认不加密，安全需要考虑HTTPS协议传输。
