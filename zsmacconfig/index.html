包管理工具

包管理模块用于解决代码包的下载管理与依赖关系，常见于从服务器下载第三方包和命令行程序，以及开源个人代码。

npm

  • NPM CLI Commond Line Interface => PowerShell & Terminal

与 GUI 图形化操作界面对应的 CLI 通过键盘录入指令，不支持鼠标操作。

  • 通过 npm 安装包到生产|开发|全局环境

生产环境依赖存放于 dependencies => 项目插件 ElementUI、Echarts 等。

开发环境依赖储存于 devDependencies => 程序运行时并不需要的 gulp、webpack 等用来压缩，打包代码的工具。

# 安装包到生产环境 dependencies 中
$ npm i _module_ -S
$ npm install _module_ --save
# 安装包到开发环境 devDependencies 中
$ npm i _module_ -D
$ npm install _module_ --save-dev
# 安装包到全局环境
$ npm i _module_ -g
  • npm 依赖包操作

从 npm@5 开始，通过将完整性不匹配视为缓存未命中，缓存可以从损坏问题中自我修复，因此从缓存中提取的数据保证有效。删除缓存只会让包管理器运行效率降低，且不能纠正遇到的任何问题。

# 查看某个 package 的最新版本|在 npm 服务器上所有发布过的版本
$ npm view <packageName> version|versions # 具体版本
$ npm info <packageName> # 更加详细
# npm 查看本地安装的包版本号
$ npm ls <packageName> # 本地包
$ npm ls <packageName> -g # 全局安装包
# 查看本机 npm 缓存的位置
$ npm config get cache
# 清除 npm 缓存
$ npm cache clean -f
  • package(-lock).json

通常不会将 node-modules 文件夹提交 Git，因通过 npm install 安装包时，可能出现版本不兼容、补丁或次版本缺陷等问题。而这也是 package.json 的特殊弊端。在 NPM5 引入的 package-lock.json 文件可以跟踪并固化被安装的每个软件包的确切版本，以便包模块以相同的方式被 100％ 复制。

当 npm 包某些功能布新的版本，通常的做法是直接去修改 package.json 到指定版本。package-lock.json 文件中 integrity 用于校验软件包；resolved 指向软件包的位置。

在 package.json 中，可以使用 semver 表示法设置要升级到的版本。版本号严格按照 "主版本号.次版本号.修订号" 格式命名。当某个版本改动比较大、并非稳定而且可能无法满足预期的兼容性需求时，会先发布一个先行版本。先行版本号可加在版本号的后边，即先加上一个连接号再加上一连串以句点分隔的标识符和版本编译信息。常见先行版本应说明内部版本 alpha、公测版本 beta、正式版的候选 rc Release candiate。主版本号一般作为有重大 api 改变，往往不向下兼容，次版本号也为小版本号，常新增功能，也向下兼容，修订号为补丁号，通常修复一些 bug。npm 安装包时允许使用一些特殊符号表示安装的版本范围。

// 先行版本号
a.b.c-beta.d
// 取最新的c的版本号值,a与b保持不变
~a.b.c
// 取b和c均为最新版本号,a保持不变
^a.b.c
// 始终使用确切的版本
a.b.c

如果自行发布包，在修改模块后需要更新发布版本，可以通过 Semver 规范的命令来完成，避免版本混乱。

npm version patch => 升级修订版本号
npm version minor => 升级次版本号
npm version major => 升级主版本号
  • .npmrc => npm 用户配置文件

npm 从命令行、环境变量和 .npmrc 文件获取配置设置。使用 npm config 相关命令可更新和编辑用户与全局 .npmrc 文件的内容。

# 添加
$ yarn add [package]
$ yarn add [package]@[version]

Bugs 解决

  • 开启终端出现提示 => insecure directories
zsh compinit: insecure directories, run compaudit for list. Ignore insecure directories and continue [y] or abort compinit [n]?

解决方式是通过列出不安全的目录后，将对应的目录赋与权限。

$ compaudit # 列出不安全目录
/usr/local/share/zsh
$ cd /usr/local/share/
$ sudo chmod -R 755 ./zsh # 对应的目录赋与权限
  • [WARNING]: Console output during zsh initialization detected => Issues

可于 p10k configure 或手动抑制此警告定义，需要在 ~./zshrc 中定义参数变量。

typeset -g POWERLEVEL9K_INSTANT_PROMPT=quiet

$ echo $POWERLEVEL9K_INSTANT_PROMPT

Bugs Handling

打开 txt 文件出现中文乱码

进入 TextEdit 的 Preferences 面板，点击 Open and Save 页签，在 Plain Text File Encoding 内的 Opening files 选择 Chinese GB 18030，乱码问题即可解决。

中文编码的 txt 文件打不开

"The document “.txt” could not be opened. Text encoding Chinese (GB 18030) isn’t applicable."

  • 对于内容较少的可以拖入浏览器，依靠其自动识别编码打开
  • iconv -c -f GB18030 -t UTF-8 [INPUTFILE...] >> [新文件名称]

按键失灵

Mac 外放没有声音

mbp 被迫喝咖啡后存在渗入腐蚀风险，在完成硬件清理后发现无法外放声音。

通过蓝牙连接以及 3.5mm 耳机测试还是存在声音，排除硬件模组受损。但是对于有线耳机会存在电流声，通过查阅相关的问题，主要可能是系统更新所导致。

=> 如果 Mac 內置喇叭無法正常運作
从 Apple 选单  中，选择「系统偏好设定」。按一下「声音」。按一下「输出」分页，然后选择内置喇叭。如果「输出」音量滑杆设定过低，请将滑杆向右移，然后再测试一次。如有勾选「静音」选框，请取消勾选，然后再测试一次。

Code Error

  • 进行 Homebrew 更新时，出现错误阻止了更新
shell-init: error retrieving current directory: getcwd: cannot access parent directories: No such file or directory

getcwd 全称 => get current working directory。

出现此错误的原因通常是无法定位到当前的工作目录，即系统隐藏文件内概率不能定位路径操作。
